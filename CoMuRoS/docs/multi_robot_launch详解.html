multi_robot.launch.py 详解

多机器人仿真启动的"总开关"——Ignition Gazebo 多机器人系统启动入口 · 226 行

一、文件概览

multi_robot.launch.py 是 CoMuRoS 系统的仿真启动入口。它负责在 Ignition Gazebo（现名 Gazebo Harmonic）中一次性拉起整个多机器人仿真环境——包括仿真引擎、物理世界、3 台异质机器人、通信桥接和可视化工具。

一句话核心逻辑

启动 Gazebo 仿真服务器 → 分时延迟启动 3 台机器人（无人机、2 台全向车）→ 桥接 ROS 2 ↔ Gazebo 通信 → 可选 RViz 可视化

核心设计思路：使用 TimerAction 实现分时启动（Staggered Launch），避免多个机器人同时加载 SDF 描述文件导致 CPU/内存资源竞争。机器人 3（无人机）5 秒后启动，机器人 2 和 机器人 1 分别间隔 10 秒依次启动。
属性值
文件路径robots/multi_robot/launch/multi_robot.launch.py
仿真平台Ignition Gazebo（Gazebo Harmonic）
世界文件food_court.sdf（美食广场场景）
机器人总数3 台（异构：1 无人机 + 2 全向车）

三、启动参数（Launch Arguments）

文件顶部（第 40-54 行）声明了 5 个启动参数，控制仿真和可视化的行为：

参数名默认值类型用途

参数传递路径

这些参数不仅在本文件中使用，还会通过 launch_arguments 传递给子 launch 文件（如 yahboom_robot.launch.py 和 x3_uav_robot.launch.py），实现参数级联。

四、Gazebo 仿真引擎启动

仿真环境的启动分为服务端和客户端两个部分（第 71-96 行），这是 Ignition Gazebo 的标准架构：

4.1 Gazebo 服务端（第 71-80 行）

start_gz_server_cmd = IncludeLaunchDescription(
    PythonLaunchDescriptionSource(
        os.path.join(pkg_ros_gz_sim, 'launch', 'gz_sim.launch.py')
    ),
    launch_arguments={
        'gz_args': [ '-r -s -v2 ', world_file ],
        'on_exit_shutdown': 'true'
    }.items(),
    condition=IfCondition(use_ignition)
)
参数含义
-r自动开始仿真（Auto-Start），加载后立即运行
-s服务端模式（Server Mode），不启动 GUI
-v2日志级别 Verbose 2，输出详细的调试信息
on_exit_shutdownROS 2 退出时自动关闭 Gazebo 进程

4.2 Gazebo 客户端（第 83-96 行）

start_gz_client_cmd = IncludeLaunchDescription(
    PythonLaunchDescriptionSource(
        os.path.join(pkg_ros_gz_sim, 'launch', 'gz_sim.launch.py')
    ),
    launch_arguments={
        'gz_args': '-g -v2'
    }.items(),
    condition=IfCondition(
        AndSubstitution(
            NotSubstitution(headless),
            use_ignition
        )
    )
)
客户端启动条件

客户端（GUI）仅在非无头模式且use_ignition 为 true 时启动。即 headless:=false（默认）时用户可以看到 3D 仿真画面；设置为 headless:=true 则只在服务器端运行，适合服务器部署或批量测试。

服务端/客户端分离架构：Ignition Gazebo 的 SDF 物理引擎在服务端运行，渲染和用户交互在客户端。这种分离允许在远程服务器上运行仿真，本地仅连接客户端查看。这对于 Docker 部署或 CI/CD 测试特别有用。

五、时钟与 TF 桥接节点

第 99-109 行创建了一个 ros_gz_bridge 的 parameter_bridge 节点，负责 ROS 2 与 Ignition Gazebo 之间的通信转换：

igntion_bridge_cmd = Node(
    package='ros_gz_bridge',
    executable='parameter_bridge',
    name='clock_tf_bridge',
    arguments=[
        '/clock@rosgraph_msgs/msg/Clock[gz.msgs.Clock',
        '/tf@tf2_msgs/msg/TFMessage[gz.msgs.Pose_V',
    ],
    parameters=[{'use_sim_time': use_sim_time}],
    output='screen',
)
话题ROS 2 类型Gazebo 类型方向用途
/clockrosgraph_msgs/Clockgz.msgs.Clock[ Gazebo → ROS同步仿真时间到所有 ROS 2 节点
/tftf2_msgs/TFMessagegz.msgs.Pose_V[ Gazebo → ROS将 Gazebo 中的位姿信息发布为 ROS TF 树
命名注意：代码中变量名为 igntion_bridge_cmd（拼写错误 "igntion" 而非 "ignition"），这在 Python 语法上没有问题，但反映了该文件在快速开发过程中的痕迹。同样，第 46 行的 decalre_use_headless_arg 和第 52 行的 decalre_multi_rviz_cmd 也存在拼写不一致的情况。
为什么需要桥接？ROS 2 的节点（如机器人控制器、LLM 节点）需要感知仿真时间和机器人之间的坐标变换。/clock 让所有节点使用统一的仿真时间，保证指令和传感器数据的时间戳一致；/tf 让导航、可视化等模块知道每个机器人在世界坐标系中的位置。

七、TimerAction 延迟启动机制

TimerAction 是 ROS 2 Launch 系统提供的一个关键工具（第 25 行引入），它包装一组 action，在指定延迟后执行。其设计模式如下：

TimerAction 模式
TimerAction(
    period=延迟秒数,
    actions=[
        IncludeLaunchDescription(
            PythonLaunchDescriptionSource(launch_file_path),
            launch_arguments={...}.items()
        )
    ]
)
封装 launch 文件。将要延迟启动的机器人 IncludeLaunchDescription 放入 actions 列表。
指定延迟时间。period 参数为该 TimerAction 相对于整个 launch 文件开始执行时刻的延迟秒数。
立即注册。TimerAction 在 LaunchDescription 被解析时立即注册到事件循环中，但 actions 在 period 秒后才执行。
自动触发。无需手动管理，ROS 2 Launch 系统内部维护定时器队列，到期自动执行。
为什么不是 sleep？新手可能会想到用 time.sleep(5) 实现延迟，但这会阻塞整个 launch 进程——在 sleep 期间，启动系统无法处理其他事件（如 ROS 2 通信、退出信号）。TimerAction 是异步非阻塞的，注册后立即返回，launch 系统继续执行后续内容。

十一、RViz 可视化

第 199-207 行提供了一个可选的单窗口 RViz 节点：

single_rviz_node = Node(
    package='rviz2',
    executable='rviz2',
    name='single_rviz',
    arguments=['-d', os.path.join(pkg_shared, 'rviz', 'multi_robot.rviz')],
    parameters=[{'use_sim_time': use_sim_time}],
    condition=IfCondition(use_single_rviz),
    output='screen'
)
RViz 配置
  • 配置文件：multi_robot/rviz/multi_robot.rviz —— 预配置了所有 3 台机器人的 TF 显示和模型可视化
  • 条件执行：只有 use_single_rviz:=true 时才会启动
  • 仿真时间：使用 use_sim_time 参数，与仿真时间同步
双重 RViz 机制：启动文件同时支持两种 RViz 模式：use_single_rviz 在此文件中直接创建节点；use_multi_rviz 作为参数传递给子 launch 文件，由每个机器人各自启动独立的 RViz 窗口。用户可以同时启用两者（虽然通常只用一个）。

十二、LaunchDescription 组合

第 209-226 行将所有组件组装到最终的 LaunchDescription 中：

return LaunchDescription([
    # 参数声明
    declare_use_sim_time,
    declare_use_ignition,
    decalre_use_headless_arg,
    decalre_multi_rviz_cmd,
    declare_single_rviz_cmd,

    # Gazebo 仿真引擎
    start_gz_server_cmd,
    start_gz_client_cmd,

    # ROS 2 ↔ Gazebo 桥接
    igntion_bridge_cmd,

    # 三台机器人（分时启动）
    robot3_launch,    # T=5s, 无人机
    robot2_launch,    # T=15s, 全向车 1
    robot1_launch,    # T=25s, 全向车 2

    # 可视化
    single_rviz_node
])
执行顺序

LaunchDescription 是一个有序列表，ROS 2 Launch 系统按顺序处理每个元素：

  1. 先声明参数—— 所有 DeclareLaunchArgument 必须放在最前面，确保后续 Action 可以引用这些参数
  2. 再启动 Gazebo 引擎—— 仿真环境必须先就绪，机器人才能在其中生成
  3. 桥接节点紧随其后—— 在机器人加入之前建立通信通道
  4. 分时启动机器人—— 三个 TimerAction 立即注册，但 actions 在指定延时后执行
  5. 最后是可选的 RViz—— 可视化工具通常最后启动

十三、核心要点总结

关键认知
  • TimerAction 是分时启动的关键。利用 ROS 2 Launch 的异步定时器机制，在不阻塞启动进程的前提下实现机器人的延迟生成，有效避开资源竞争。
  • 异构机器人、统一启动入口。通过 IncludeLaunchDescription 和参数传递，一个启动文件可以灵活组合多个功能包中的独立 launch 文件，实现异构机器人的统一编排。
  • 命名空间隔离。每台机器人通过 prefix 参数（r1/r2/r3）获得独立的 ROS 2 命名空间和 TF 前缀，避免了多机器人系统中的话题和坐标变换冲突。
  • Gazebo 服务端/客户端分离。支持无头模式（headless），适合服务器部署和自动化测试场景，GUI 和物理引擎可独立运行。
  • 参数级联机制。顶层参数（如 use_sim_time、use_ignition）穿透传递给子 launch 文件，形成层级化的配置体系。
  • 条件执行（IfCondition）。Gazebo 客户端和 RViz 窗口都通过 IfCondition 条件控制，用户可以在启动时灵活选择启用哪些组件。
