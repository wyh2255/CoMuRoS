## 一、ROS 2 核心概念

### 1. ROS 2 Service（请求-响应模式）

- **Topic** = 广播电台，发布者不管谁在听，订阅者只管听
- **Service** = 打电话，客户端请求 -> 服务端响应，一次通话一次回答
- **三要素**：.srv 接口定义（请求/响应结构）-> 服务端（Server，干事的人）-> 客户端（Client，派活的人）

项目中 Service 定义在 robot_interface/srv/：

| Service | 服务端 | 客户端 |
|---------|--------|--------|
| GotoPoseHolonomic | holonomic_position_controller | cleaning_bot_llm |
| GotoPoseDrone | drone_position_controller | drone_llm |
| StartPick | 未实现（可能由仿真提供） | robot_llm |

### 2. ROS 2 Callback（回调）

- 回调 = "当某事件发生时，ROS 自动调用的函数"
- rclpy.spin(node) = 无限循环，持续监听事件并分发给对应的回调
- **四种回调类型**：
  - 订阅回调（create_subscription）-> 话题有新消息
  - 服务回调（create_service）-> 收到服务请求
  - 定时器回调（create_timer）-> 每 N 秒触发
  - 客户端响应回调（add_done_callback）-> 服务端返回结果
- 没有 spin，节点就像不接电话的人——注册了但永远不会处理

### 3. 注册回调的方式

```python
# 订阅回调
self.sub = self.create_subscription(String, '/topic', self.callback, 10)

# 服务端回调
self.srv = self.create_service(GotoPoseHolonomic, '/goto_pose', self.callback)

# 定时器回调
self.timer = self.create_timer(1.0, self.callback)

# 客户端响应回调
future = self.client.call_async(request)
future.add_done_callback(self.callback)
```

## 二、task_manager.py（指挥官）

**一句话**：订阅对话 -> 调用 LLM 解析任务 -> 发布结构化 JSON 给各机器人 LLM

### 关键认知

- **不做底层控制**，只负责"想清楚要做什么"
- **事件驱动**：收到人类消息 -> 调用 LLM -> 发布 JSON。收到 COMPLETE -> 发下一个任务。收到 Event -> 重规划
- **双任务模式**：Plan（顺序执行，队列逐个发布）和 Independent Tasks（并行执行，批量发布）
- **状态全在内存**：重启即丢失（设计取舍：简单 > 持久）
- **4 大类 LLM 后端**：OpenAI、Gemini、xAI(Grok)、Ollama，通过 model 参数切换
- **核心弱点**：高度依赖 LLM 输出格式一致性，正则解析脆弱

### 数据流

```
用户输入 cleanup
  -> /chat/input -> chat_manager -> /chat/output
    -> task_manager.on_chat_output
      -> parse_tasks_with_gpt()
        -> LLM API（GPT / Gemini / Ollama）
          -> publish_tasks_json() -> /task_manager/tasks_json
            -> 各机器人 LLM 节点
```

## 三、cleaning_bot_llm.py（执行者）

**一句话**：订阅任务 JSON -> LLM 生成执行代码 -> 调用 ROS 2 Service 移动机器人 -> 发布完成状态

### 关键认知

- **LLM 出代码，节点来执行**：GPT-4o 动态生成 Python 代码 -> exec() 在受限沙箱中执行
- **task_manager 的"干活的手"**：一个总体规划，一个具体执行
- **Service 调用是可取消的**：协作式取消标志位 + 循环中持续检测
- **多线程执行器**（MultiThreadedExecutor 4 线程）：因为 Service 调用阻塞，必须让其他回调有执行机会
- **受限沙箱**：exec(code, {"__builtins__": {}}, {"node": node}) 禁用内置函数
- **回调组设计**：MutuallyExclusive（互斥）和 Reentrant（可重入）搭配使用

### 任务执行流程

```
on_tasks_json 收到 JSON
  -> 解析匹配机器人名称
  -> 检查 STOP 指令 -> 调用 stop_tasks()
  -> 重置取消标志
  -> execute_task()
    -> 读取聊天历史
    -> 构建 system prompt（能力 + 状态 + 可用动作）
    -> GPT-4o 生成 Python 代码
    -> execute_python_code() 通过 exec() 执行
    -> 发布 COMPLETED 状态
```

## 四、全向位置控制器（holonomic_position_controller_service.py）

**一句话**：接收 GotoPoseHolonomic 服务请求 -> P 控制计算速度 -> 发布 /cmd_vel -> 到达返回

### 核心信息

- **行数**：253 行 | **控制算法**：P 控制（比例控制）
- **控制维度**：2D（x, y）+ 偏航角 | **收敛阈值**：位置 0.05m，偏航 0.05rad
- **控制增益**：kp_xy=1.2，kp_yaw=2.0，速度限幅 [-1.0, 1.0]
- **坐标变换**：无（世界坐标系直接控制）

### 阻塞式服务回调模式

服务回调不立即返回，而是在回调内执行 while 循环进行实时控制。这需要：
- **ReentrantCallbackGroup** — 不阻塞其他回调的执行
- **MultiThreadedExecutor(4)** — 有额外线程处理 odom 订阅回调

没有这两者的配合，odom 数据永远无法更新，控制循环会陷入死局。

### 三种退出路径

| 条件 | 触发条件 | 结果 |
|------|---------|------|
| 到达目标 | 距离 < 0.05m 且 偏航误差 < 0.05rad | success=True |
| 取消 | cancel_requested=True | success=False |
| 超时 | 时间超过 ~100,000 秒（等价于无限） | success=False |

## 五、无人机位置控制器（drone_position_controller_service.py）

**一句话**：接收 GotoPoseDrone 服务请求 -> 三步 PID 控制（先到位 -> 再转向 -> 后微调）-> 发布 /cmd_vel

### 核心升级（对比全向控制器）

| 维度 | 全向控制器 (253 行) | 无人机控制器 (579 行) |
|------|-------------------|---------------------|
| 控制算法 | P 控制 | **PID** 比例-积分-微分 |
| 控制策略 | 一步到位 | **三步递进** |
| 空间维度 | 2D (x, y) | **3D (x, y, z)** |
| 坐标变换 | 无 | **世界→机体旋转变换** |
| 积分项 | 无 | 有（含抗积分饱和） |
| 微分项 | 无 | 有（含 dt 计算） |

### PIDController 类

封装了位置式 PID 算法，三个关键设计：
- **抗积分饱和**：integral_limit 限制积分累积量，防止超调
- **输出限幅**：output_limit 限制最大控制输出
- **dt 保护**：首次循环 dt=0 时跳过微分项，避免除零

四轴独立 PID 参数：

| 轴 | Kp | Ki | Kd | 设计思路 |
|----|----|----|----|---------|
| X/Y（水平） | 2.5 | 0.1 | 0.8 | 高比例快响应，微分防震荡 |
| Z（高度） | 2.5 | **0.2** | 0.6 | 更高积分增益消除重力导致的静差 |
| Yaw（偏航） | 1.0 | 0.1 | 0.5 | 较低增益避免偏航震荡 |

### 三步递进控制策略

```
Step 1: move_to_position   只控 X/Y/Z，偏航速度=0     → 先到附近
Step 2: correct_yaw        锁定当前位置，只转偏航     → 角度对准
Step 3: fine_tune_both     同时控制四轴，精细微调     → 最终精度
```

每步失败则立即停止并返回。Step 2 中偏航角速度额外 ×0.5 防震荡。

### 世界→机体坐标变换

因为 `/cmd_vel` 使用机体坐标系（linear.x=机头方向），而 PID 计算的误差在世界坐标系下，必须用旋转矩阵进行变换：

```
[vx_body]   [cos(yaw)  sin(yaw)] [vx_world]
[vy_body] = [-sin(yaw) cos(yaw)] [vy_world]
```

### start_drone()

发送 angular.z=0.1 的微小扰动激活 Gazebo 仿真中的电机，然后停止。地面机器人不需要这个步骤。

## 六、控制层的共同设计模式

两个位置控制器共享相同的架构设计：

- **阻塞式服务回调** + ReentrantCallbackGroup + MultiThreadedExecutor
- **协作式取消**通过 cancel_requested 标志位
- **角度归一化** wrap() = atan2(sin(a), cos(a))
- **底层执行层定位**：都是 LLM 节点的"手"，说去哪儿就去哪儿
- **相同接口模式**：订阅 odom + 订阅 cancel + 发布 cmd_vel + 服务 goto_pose

## 七、主包 vs 备选包对比（第三轮核心发现）

项目存在三组"主包 vs 备选包"的代码分化，均服务于同一台机器人，但处于不同的包路径下。

### 7.1 总体格局

| 机器人 | 主包（在用） | 备选包（robots/下） | 启动配置中的证据 |
|--------|------------|-------------------|----------------|
| 清洁机器人 | `cleaning_bot/` | `robots/yahboom/yahboom_llm/` | `robot_llms.launch.py` → `package='cleaning_bot'` |
| 无人机 | `drone/` | `robots/x3_uav/x3_uav_llm/` | `robot_llms.launch.py` → `package='drone'` |

**关键发现**：`robot_services.launch.py` 中 r1 和 r2 使用了**同一个** `cleaning_bot` 可执行文件，仅通过 `namespace` 参数区分（r1 / r2），说明一个控制器二进制可服务多个机器人实例。

### 7.2 LLM 节点差异总结

| 维度 | 主包（cleaning_bot / drone） | 备选包（yahboom_llm / x3_uav_llm） |
|------|---------------------------|-----------------------------------|
| **取消机制** | ✅ **双通道**：`_task_cancelled` 标志 + Bool话题 + `TaskCancelledException` 异常 | ❌ **单通道**：仅 Bool 话题通知控制器，LLM 层无中断 |
| **`exec()` 上下文** | 传入 `node` + `TaskCancelledException` | 只传 `node` |
| **`check_cancelled()`** | ✅ `goto_service` 循环和 `clean()` 航点中持续检测 | ❌ 没有 |
| **`clean()`/`hover()` 状态追踪** | **丰富**：多步 `update_robot_state()` 记录进度 | **极简**：直接调 service + 更新状态 |
| **行数** | 929 / 965 行 | 712 / 768 行 |

### 7.3 位置控制器差异总结

| 维度 | drone/（主包无人机） | x3_uav_llm/（备选无人机） |
|------|-------------------|------------------------|
| **控制策略** | **三步 PID**（move → yaw → fine-tune） | **单步 P/PI** |
| **PID 类** | ✅ 独立 `PIDController` 类 | ❌ 控制逻辑内联 |
| **坐标变换** | ✅ 世界→机体旋转变换 | ❌ 无 |
| **代码量** | 579 行 | 234 行 |

**全向控制器**则几乎相同（都是 P 控制，kp_xy=1.2, kp_yaw=2.0），仅默认 namespace 不同（r1 vs r2）。

### 7.4 为什么 yahboom/ 和 x3_uav/ 存在？

这两个包在 `robots/` 模块中充当了**不同角色**：

```
robots/yahboom/
├── yahboom_llm/                   ← 备选 LLM 节点（未启用）
├── yahboom_rosmaster_description/ ← 正式使用：URDF + 网格 + RViz
└── yahboom_rosmaster_gazebo/      ← 正式使用：worlds + config + launch

robots/x3_uav/
├── x3_uav_llm/                    ← 备选 LLM + 控制器（未启用）
├── x3_uav_ignition/               ← 正式使用：bridge config + launch
└── x3_uav_description/            ← 正式使用：URDF + Gazebo 插件
```

**结论**：LLM 和控制器节点在主包中更成熟，`robots/` 下的是简化版。但描述模型和 Gazebo 配置只在 `robots/` 中，被仿真启动系统使用。
