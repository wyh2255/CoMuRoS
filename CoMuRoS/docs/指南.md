# CoMuRoS 学习指南

## 项目概览一句话

CoMuRoS = **Co**llaborative **Mu**lti-**R**obot **S**ystem：通过自然语言（中文/英文）指挥异构机器人团队完成任务的 ROS 2 系统，使用大语言模型（LLM）进行任务规划与事件驱动重规划。

---

## 学习路线（共四个阶段）

### 阶段一：理解全局架构

**目标**：知道系统有哪些组件、它们之间如何通信。

#### 1.1 读顶层文档

| 文件 | 用途 |
|------|------|
| `README.md` | 项目概述、快速启动、特性列表 |
| `CLAUDE.md` | 架构总结、包结构、数据流图 |

#### 1.2 读两个最重要的 launch 文件

**chat_system.launch.py** (`chatty/launch/`)

启动以下节点（了解即可，后面逐个深入）：
| 节点 | 职责 |
|------|------|
| `chat_gui` | 桌面聊天窗口（customtkinter） |
| `chat_manager` | 消息路由、会话历史管理 |
| `task_manager` | **核心节点** — 调用 LLM 进行任务规划 |
| `time_publisher` | 发布仿真时间到 `/current_time` |
| `microphone` / `speak` / `listen` | 可选的音频输入输出 |

**robot_llms.launch.py** (`robot_llm/launch/`)

启动所有机器人的 LLM 接口节点：
- `cleaning_bot_llm_node` — 清洁机器人
- `delivery_bot_llm_node` — 配送机器人
- `drone_llm_node` — 无人机

#### 1.3 理解整体数据流

```
用户输入（GUI 或语音）
  →  /chat/input 话题
    →  ChatManager（路由、存历史）
      →  /chat/output 话题
        →  TaskManager（LLM 将自然语言解析为结构化任务）
          →  /task_manager/tasks_json 话题
            →  各 Robot LLM 节点（各自订阅）
              →  ROS 2 Service（GotoPoseHolonomic、GotoPoseDrone 等）
                →  Gazebo 仿真（Ignition Gazebo）
```

### 阶段二：追踪一条完整消息（从 GUI 到仿真）

按以下顺序读文件，每次只关注核心逻辑，忽略细节。

#### 2.1 chat_gui.py — 用户接口

- 关注点：它发布什么话题、订阅什么话题
- 入口函数 `main()` 启动 ROS 节点和 customtkinter 窗口
- 核心话题：
  - 发布 → `/chat/input`（用户输入的文字）
  - 订阅 ← `/chat/output`（系统回复和历史）
  - 订阅 ← `/current_time`（仿真时间显示）

#### 2.2 chat_manager.py — 消息中枢

- 订阅 `/chat/input`，收到后记录到历史文件，再发布到 `/chat/output`
- 提供 `get_chat_history` 服务（Trigger），供其他节点拉取完整对话
- 理解这一点：chat_manager 不涉及任何 LLM 调用，只是纯消息路由

#### 2.3 task_manager.py — 核心节点（LLM 任务规划）

这是整个系统最复杂的文件，建议分三次读：

| 读次 | 关注点 |
|------|--------|
| 第一次 | 类结构、订阅/发布的话题、LLM 模型选择（openai/gemini/xai/ollama） |
| 第二次 | LLM prompt 模板 —— 理解如何让 LLM 输出结构化的机器人任务 |
| 第三次 | "Event:" 事件驱动的重规划机制、多步骤任务序列处理 |

关键理解：
- TaskManager **不直接控制机器人**，它解析 LLM 的文本输出为 JSON 后发布
- `robot_state` 跟踪每个机器人的当前状态，传给 LLM 作为上下文
- 收到 "Event:" 前缀的消息时，触发全量重规划

#### 2.4 选一个机器人 LLM 节点阅读

推荐读 `cleaning_bot_llm.py`（`cleaning_bot/cleaning_bot/`）：

- 比 task_manager 简单得多
- 订阅 `/task_manager/tasks_json`
- 根据任务类型调用不同的 ROS 2 Service
- 理解"机器人端 LLM"和"中央 TaskManager LLM"的分工：
  - TaskManager：总体规划（谁做什么）
  - 机器人 LLM：具体执行（怎么走到那个点）

#### 2.5 理解底层接口定义

读 `robot_interface/` 目录下的文件：

| 文件 | 含义 |
|------|------|
| `srv/GotoPoseHolonomic.srv` | 全向移动机器人去目标位姿 |
| `srv/GotoPoseDrone.srv` | 无人机去目标位姿 |
| `srv/GoTo.srv` | 差速驱动机器人导航 |
| `srv/Find.srv` | 查找物体 |
| `srv/StartPick.srv` | 启动抓取 |
| `action/PickObject.action` | 抓取物体（ROS 2 Action，带反馈） |

#### 2.6 读一个控制器实现

读 `holonomic_position_controller_service.py`：

- 这是一个 ROS 2 Service 节点
- 接收目标位姿 → 转化为速度指令 → 发布到 Gazebo
- 理解 PID 控制或纯追踪如何让机器人移动到位

### 阶段三：理解仿真环境与机器人扩展

#### 3.1 仿真启动流程

读 `robots/multi_robot/launch/multi_robot.launch.py`：

- 启动 Ignition Gazebo + 食物城市场景
- 3 个机器人错峰生成（r1: 5秒, r2: 15秒, r3: 25秒）
- 配置 ROS-Gazebo 桥接（/clock, /tf）

#### 3.2 机器人模型包结构

```
robots/
├── multi_robot/         # 多机器人仿真启动
├── x3_uav/              # X3 无人机
│   ├── x3_uav_description/   # URDF 模型描述
│   ├── x3_uav_ignition/      # Ignition 仿真配置
│   └── x3_uav_llm/           # LLM 接口 + 飞行控制器
└── yahboom/             # Yahboom Rosmaster X3 地面机器人
    ├── yahboom_rosmaster_description/  # URDF
    ├── yahboom_rosmaster_gazebo/       # Gazebo 仿真
    └── yahboom_llm/                    # LLM 接口 + 运动控制器
```

每个机器人包都遵循描述 → 仿真 → LLM 的三层模式。

#### 3.3 命名空间约定

| 命名空间 | 机器人类型 |
|----------|-----------|
| `/r1/` | 清洁机器人（全向移动） |
| `/r2/` | 配送机器人（全向移动） |
| `/r3/` | 无人机 |

### 阶段四：深入 LLM 与配置

#### 4.1 多 LLM 支持

`task_manager.py` 顶部有一个模型列表，通过 `model` 参数索引选择：

- 0-3: OpenAI GPT 系列
- 4-6: Gemini 系列
- 7-8: XAI Grok 系列
- 9: Ollama 本地模型
- 10+: 其他版本

API 密钥从环境变量读取：`OPENAI_API_KEY`, `GEMINI_API_KEY`, `XAI_API_KEY`

#### 4.2 机器人配置文件

`chatty/config/robot_config_*.json` 定义了：

```json
{
  "robot_names": ["cleaning_bot", "delivery_bot", "drone"],
  "robot_capabilities": {
    "cleaning_bot": "可以清洁地面、移动到指定位置"
  },
  "robot_morphology": {
    "cleaning_bot": "全向移动底盘+清洁模块"
  },
  "robot_states": {
    "cleaning_bot": {
      "relative_position": [11.0, 0.0, 0.05],
      "mode": "Idle"
    }
  }
}
```

这些信息被拼接进 LLM prompt，让大模型了解机器人团队的能力边界。

## FAQ

**Q: 系统依赖哪些 ROS 2 包？**

每个包的 `package.xml` 列出其依赖。基础依赖：`rclpy`, `std_msgs`, `std_srvs`。仿真依赖：`ignition-gazebo` 相关桥接包。

**Q: LLM prompt 在哪里？**

`task_manager.py` 的 `get_gpt_response()` 方法中拼接了完整的 prompt 字符串，包含机器人能力、当前状态、任务历史。

**Q: 如何调试 LLM 输出？**

TaskManager 会将 LLM 的原始回复和解析后的 JSON 都记录到日志中。可以通过 `ros2 topic echo /task_manager/tasks_json` 实时观察。

**Q: 如何添加新的 LLM 后端？**

在 `task_manager.py` 的模型列表中添加新的模型配置，并在 `get_gpt_response()` 中添加相应的 API 调用分支。
