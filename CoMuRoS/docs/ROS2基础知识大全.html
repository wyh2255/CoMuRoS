一、ROS 2 Node 基类（rclpy.node.Node）

一句话：Node 是 ROS 2 Python 节点的"入场券"——不继承它，你的类无法与 ROS 2 系统通信。

1.1 什么是 Node 基类？

rclpy.node.Node 是所有 ROS 2 Python 节点的基类（base class）。它来自 rclpy（ROS Client Library for Python），本质上是一个 Python 类，封装了节点与 ROS 2 系统通信所需的全部基础设施。

核心公式

class MyNode(Node):
    def __init__(self):
        super().__init__("my_node_name")

1.3 不调用 super().__init__() 会怎样？

崩溃。 虽然你写了 class MyNode(Node)，但节点名称没有注册，内部通信层没有初始化。 任何调用 self.create_publisher() 的代码都会抛出异常。

super().__init__("name") 实际干了三件事：

  1. 注册节点——告诉 ROS 2 系统"我叫 xxx，我上线了"，ros2 node list 能看见它
  2. 初始化通信层——准备好发布器、订阅器、服务、客户端等底层设施
  3. 激活日志系统——self.get_logger().info("msg") 才能正常工作

1.4 没有 spin 会怎样？

回调函数（callback）注册好了，但如果没有 rclpy.spin(node)，节点就像不接电话的人——注册了但永远不会处理任何消息。

💡 理解 spin： rclpy.spin(node) = 一个无限循环，持续监听 ROS 2 网络上的事件，分发给对应的回调函数。 没有 spin，订阅了话题也不会收到消息，注册了服务也不会响应请求。 
rclpy.spin(node) 的伪代码：
while True:
    for event in get_next_ros_event():
        if event.type == "subscription":
            event.callback(event.msg)
        elif event.type == "service_request":
            event.callback(event.request, event.response)
        elif event.type == "timer":
            event.callback()
        elif event.type == "service_response":
            event.done_callback(event.future)

1.6 最小完整示例

#!/usr/bin/env python3
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class MinimalPublisher(Node):
    def __init__(self):
        super().__init__('minimal_publisher')  # 必须
        self.pub = self.create_publisher(String, '/topic', 10)
        self.timer = self.create_timer(1.0, self.timer_callback)

    def timer_callback(self):
        msg = String()
        msg.data = "Hello ROS 2"
        self.pub.publish(msg)

def main():
    rclpy.init()
    node = MinimalPublisher()
    rclpy.spin(node)  # 没有 spin 就不会触发 timer_callback
    node.destroy_node()
    rclpy.shutdown()

二、LaunchConfiguration（启动配置变量）

一句话：LaunchConfiguration = ROS 2 启动文件中的"变量占位符"——定义时只是一个名字，运行时才拿到真实值。

2.1 为什么需要它？

看这个场景：你想写一个通用的机器人启动文件，既能启动单个机器人也能启动多个机器人，位置坐标、命名空间前缀、是否启用 RViz 都要能灵活控制。不用 LaunchConfiguration，你就得写死所有值，每个场景复制一份文件。用了它，一个文件 + 不同参数 = 不同行为。

2.2 两步走：声明 vs 使用

步骤API干了什么
1. 声明 DeclareLaunchArgument('name', default_value='x3') 告诉系统"我有一个参数叫 name，没人传就用 'x3'"
2. 使用 LaunchConfiguration('name') 创建一个占位符对象，在启动真正执行时才求值
🎯 关键认知

LaunchConfiguration('prefix') 不是字符串，它是一个惰性求值对象。 你在 Python 代码里用它拼接路径、构建命令时，ROS 2 的 Launch 系统会在运行时才去查它的真实值。 这就是为什么你需要用 PythonExpression、PathJoinSubstitution 等工具来处理它，而不是直接做字符串拼接。

2.3 项目中的实际用法

在 x3_uav_robot.launch.py 中（用户选中行 84）：

# 第一步：声明参数及其默认值
prefix_arg = DeclareLaunchArgument('prefix', default_value='x3')

# 第二步：创建惰性占位符
prefix = LaunchConfiguration('prefix')  # 此时还不是字符串！

# 第三步：用在哪里？
# 1) 作为 Node 的 namespace 参数
robot_state_publisher_cmd = Node(
    namespace=prefix,                    # 惰性求值 — 运行时才确定命名空间
    ...
)

# 2) 拼接 xacro 命令参数
robot_description_content = Command([
    'xacro ', xacro_file,
    ' prefix:=', prefix,              # xacro 参数
])

# 3) 动态计算话题名（PythonExpression）
topic_with_prefix = PythonExpression([
    "'/' + '", prefix, "' + '/cmd_vel'"  # → /x3/cmd_vel 或 /r1/cmd_vel
])

2.4 在 OpaqueFunction 中获取真实值

有时你需要在 Python 函数中拿到 LaunchConfiguration 的真实字符串值。OpaqueFunction 提供了 context 参数：

def bridge_topics(context, *args, **kwargs):
    # .perform(context) 才是真正求值
    prefix = LaunchConfiguration('prefix').perform(context)
    # 现在 prefix 是真正的字符串了，比如 "r1" 或 "x3"

    # 可以用它做正常的 Python 字符串操作

2.5 参数覆盖机制

声明了 LaunchConfiguration 的参数后，可以在启动时从命令行覆盖：

# 启动时覆盖 prefix 默认值
ros2 launch x3_uav_ignition x3_uav_robot.launch.py prefix:=r3

# 覆盖多个参数
ros2 launch x3_uav_ignition x3_uav_robot.launch.py \
    prefix:=r3 use_rviz:=true spawn_x:=1.0 spawn_y:=2.0

2.7 与其他 Launch 概念的关系

概念类比关系
DeclareLaunchArgument 变量的"声明" 定名字、给默认值、写说明
LaunchConfiguration 变量的"引用" 在 Node 参数、表达式、条件中使用的占位符
PythonExpression 三元运算符 当值需要条件判断时（如空 prefix 用不同话题名）
IfCondition if 语句 用 LaunchConfiguration 控制某个动作是否执行
OpaqueFunction 自定义 Python 函数 需要拿到真实字符串值做复杂逻辑时使用
📝 本节小结： 理解 LaunchConfiguration 的三层含义：
① 它是变量的惰性占位符，不是真正的字符串
② 它在运行时才求值，让一个启动文件能适配不同场景
③ 它的值可以通过命令行 name:=value 覆盖，实现"写一次，多处用"
