robots 模块详解

CoMuRoS 机器人仿真环境与LLM控制节点的完整实现 · 多机器人异构系统

一、模块概览

CoMuRoS/robots/ 是 CoMuRoS 系统的机器人核心模块，包含了所有机器人的仿真描述、LLM智能控制节点、位置控制器服务以及多机器人启动系统的完整实现。它与上层 chatty（对话管理）和 robot_llm（元启动包）协同工作，构成完整的"自然语言→机器人动作"流水线。

一句话核心逻辑

仿真世界 + 三种异构机器人（全向清洁车、全向送货机器人、四旋翼无人机）+ LLM 驱动的大脑 + P/PID 位置控制器

核心特点：robots 模块提供的不是"预编程的机器人"，而是LLM实时生成代码驱动的机器人——每个机器人的 LLM 节点都通过 GPT-4o 将自然语言转换为 Python 代码，再通过 exec() 执行，调用底层位置控制器完成物理移动。

6.3 代码执行安全沙箱

所有 LLM 节点都使用同一套安全的代码执行机制：

def execute_python_code(code: str, node=None):
    if node is None:
        node = RobotLLMNode.get_instance()
    try:
        exec(code, {"__builtins__": {}}, {"node": node})
    except Exception as e:
        node.get_logger().error("Failed to execute generated code: %s", e)
安全沙箱的限制
  • __builtins__: {} — 禁用所有内置函数（open, exec, eval, import…）
  • 仅暴露 node 对象 — LLM 只能调用 node 上预定义的方法
  • 仅暴露 TaskCancelledException（在 cleaning_bot 中）— 支持取消信号
安全边界：这种沙箱方式能阻止 LLM 访问文件系统和执行任意代码，但不能阻止 LLM 生成长时间运行或无限循环的代码。实际的安全性依赖于 GPT-4o 的指令跟随能力和 prompt 设计的严谨性。

6.4 任务匹配逻辑（三阶回退）

三个 LLM 节点共享相同的任务键名匹配策略（on_tasks_json 回调）：

# 依次尝试三种键名匹配方式：
robot_task = robot_tasks.get(robot_name)          # "cleaning_bot"

7.1 全向机器人控制器（P 控制）

控制律
vx = kp_xy × (goal_x - current_x)     # X 方向速度
vy = kp_xy × (goal_y - current_y)     # Y 方向速度（全向移动！）
wz = kp_yaw × wrap(goal_yaw - yaw)    # 偏航角速度

全向机器人可以独立控制 X 和 Y 方向速度，这是麦克纳姆轮的核心优势——不需要先转向再前进，可以斜着走。

7.2 无人机控制器（水平 P + 高度 PI）

控制律
vx = kp_xy × (goal_x - x)             # 水平 X
vy = kp_xy × (goal_y - y)             # 水平 Y
vz = kp_z × ez + ki_z × ∫ez dt        # 高度 PI 控制（带积分限幅）
wz = kp_yaw × wrap(goal_yaw - yaw)    # 偏航

高度通道使用 PI 控制是因为无人机悬停时存在重力补偿需求——纯 P 控制会产生稳态误差。积分项消除稳态误差，积分限幅（±0.5）防止积分饱和。

超时差异的考量：全向控制器的超时设为 ~100000 秒（几乎无限），意味着清洁机器人从位置 (11.0, -3.0) 移动到 (-3.5, 3.0) 再久也会等待。无人机只有 60 秒超时——这是合理的，因为无人机在空中应尽快完成任务，长时间无法到达目标可能意味着电池耗尽或物理故障。

7.3 取消机制

两个控制器都通过订阅 /{ns}/cancel_goto_pose_goal（Bool 类型）实现取消：

def cancel_callback(self, msg):
    if msg.data:
        self.cancel_requested = True

# 在控制循环中：
if self.cancel_requested:
    self.stop()                           # 发布 Twist() → 停止
    response.success = False
    response.message = "Goal canceled."
    return response

十、添加新机器人指南

robots/README.md 提供了添加新机器人的完整步骤。以下是简化的 6 步流程：

添加到仿真。在 multi_robot.launch.py 中添加 TimerAction，通过 IncludeLaunchDescription 接入你的机器人启动文件。
创建机器人包。遵循 description + gazebo/ignition + llm 三层结构，创建 URDF 模型、传感器配置、Gazebo 插件。
添加到 LLM 启动。在 robot_llm/launch/robot_llms.launch.py 中添加你的 LLM 节点定义。
添加控制器服务。在 robot_llm/launch/robot_services.launch.py 中添加你的位置控制器服务节点。
更新配置。在 chatty/config/ 的配置 JSON 中添加新机器人的名称、能力和形态描述。
构建和测试。colcon build → 启动系统 → 用自然语言测试。
关键认知
  • 三种机器人，同一架构。清洁车、送货机器人、无人机使用完全相同的"LLM 生成代码 → exec() 执行 → Service 调控制器"模型，差异仅在于传感器配置和控制器类型。
  • LLM 是大脑，控制器是脊髓。LLM 负责"去哪里"的决策（生成代码），控制器负责"怎么去"的执行（P/PI 控制循环）。
  • 命名空间隔离是多机器人的基石。通过 prefix 参数将每个机器人的话题、服务、TF 树完全隔离，避免多机器人间的冲突。
  • 阶梯式启动解决资源竞争。5 秒间隔启动机器人，避免 Gazebo 同时加载多个复杂模型时的性能瓶颈。
  • 无人机最复杂。X3 UAV 是三种机器人中最复杂的——4 个独立电机模型、PID 控制器、双 RGB-D 相机、GPT-4.1 视觉分析。
  • 全向机器人的清洁路径是预设的。6 个定位点的清洁路径是硬编码的，LLM 只能选择 "清洁或不清洁"，不能控制具体路径。
下一步学习建议：阅读 chatty 模块的 task_manager.py——了解顶层的任务规划器如何决定"谁做什么"，这是 robots 模块的上游输入来源。
