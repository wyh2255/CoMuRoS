一、文件概览

cleaning_bot_llm.py 是 CoMuRoS 系统中清洁机器人（cleaning_bot）的智能控制节点。与 task_manager.py（指挥官角色）不同，这个节点是一线执行者——它负责接收任务、理解任务、然后实际控制机器人动起来。

一句话核心逻辑

订阅任务 JSON → LLM 生成执行代码 → 调用 ROS 2 Service 移动机器人 → 发布完成状态

核心区别：task_manager 用 LLM 做任务规划（谁做什么），cleaning_bot_llm 用 LLM 做代码生成（怎么走到指定位置）。一个规划，一个执行。

回调组的设计

self.single_group = MutuallyExclusiveCallbackGroup()
self.seq_group    = MutuallyExclusiveCallbackGroup()
self.multi_group  = ReentrantCallbackGroup()
回调组类型用途
single_group互斥on_chat_output（写入文件，独立执行）
seq_group互斥on_robot_states、on_current_time（状态更新，互斥避免竞态）
multi_group可重入on_tasks_json、Service 客户端（允许并发执行）
为什么需要回调组？ROS 2 默认 Executor 是单线程的，一个回调没执行完，其他回调就得等待。ReentrantCallbackGroup 允许 Timer 或 Service 回调并发执行，避免导航过程中的回调阻塞（例如机器人正在移动时还能收到取消指令）。

设计思路：这不是"预定义动作"的模式（hardcode 所有可能性），而是让 LLM 动态生成代码去调用已有的节点方法。这意味着只要在 prompt 中告诉 LLM 有哪些函数可用，它可以组合出任意复杂的行为序列——远比预定义的有限状态机灵活。

六、GoTo Service 调用

goto_service()（第660行）是对 ROS 2 Service 客户端的一个封装，供 LLM 生成的代码调用：

def goto_service(self, x, y, yaw_deg):
    self.check_cancelled()                    # 每次调用前检查取消
    if not self._goto_client.wait_for_service(timeout_sec=5.0):
        return False

    req = GotoPoseHolonomic.Request()
    req.x = x; req.y = y; req.yaw_deg = yaw_deg
    future = self._goto_client.call_async(req)  # 异步调用

    while rclpy.ok() and not future.done():
        self.check_cancelled()                 # 循环中持续检查取消
        time.sleep(0.05)

    res = future.result()
    return res.success
为什么要用 call_async 而非 call？同步调用 call() 内部会 spin()，导致嵌套 spin，可能造成混乱或死锁。异步调用 + 手动循环等待更安全，同时循环中嵌入取消检测，实现"可取消的 service 调用"。

七、clean 方法详解

clean()（第740行）是机器人预定义的清洁行为——沿 6 个点依次导航：

清洁路径坐标
1. (11.0, -3.0) → 2. (4.0, -3.0) → 3. (-3.5, -3.0)
4. (-3.5, 3.0) → 5. (11.0, 3.0) → 6. (11.0, 0.0)

clean() 的完整执行流程：

初始化状态。发布 cleaning_progress: "0/6"，标记任务开始。
循环导航到 6 个清洁点。每次调用 goto_service()，成功则前进。
特殊处理位置 2 的障碍物。调用 remove_cube() 移除名为 "small_cube" 的方块。
模拟清洁。每次到达后 time.sleep(2.0) 模拟清洁动作。
更新状态。每个位置完成后更新 robot_states 字典中的进度字段。
标记完成。全部 6 点完成后发布 COMPLETED 状态。
每次移动前后都调用 check_cancelled()：这是优雅取消的关键。如果外部发送 STOP 指令，self._task_cancelled 被置为 True，check_cancelled() 抛出异常，on_tasks_json 的 try/except 捕获并标记为 INTERRUPTED。

八、任务取消机制

这是文件中的一个重要设计模式。取消机制由三部分组成：

组件位置作用
_task_cancelled 标志位__init__（第109行）布尔标志，True 表示需要取消
check_cancelled()第195行检查标志位，为 True 则抛出 TaskCancelledException
stop_tasks()第489行设置标志位 + 发布取消话题消息
reset_cancellation()第201行开始新任务前清除标志位

取消的完整链路：

收到"stop"指令
→
stop_tasks()
_task_cancelled = True
→
check_cancelled()
抛出异常
→

→ INTERRUPTED
协作式取消：不是强制终止线程，而是通过标志位让执行中的代码主动检查后自行退出。好处是安全，坏处是如果 goto_service 被阻塞在 time.sleep() 中，需要等 sleep 结束才能响应取消。

九、代码执行引擎

execute_python_code()（第874行）是一个独立函数，负责执行 LLM 生成的 Python 代码：

def execute_python_code(code, node=None):
    if node is None:
        node = RobotLLMNode.get_instance()
    try:
        exec(code, {"__builtins__": {}}, {
            "node": node,
            "TaskCancelledException": TaskCancelledException
        })
    except TaskCancelledException:
        raise
安全考虑

这段 exec() 使用了受限命名空间：{"__builtins__": {}} 阻止了大部分内置函数（如 open()、os.system()）的访问。LLM 生成的代码只能访问 node（节点实例）和 TaskCancelledException。这是对 LLM 生成代码的一种基本沙箱保护。

十、main 入口

def main(args=None):
    rclpy.init(args=args)
    node = RobotLLMNode()
    executor = rclpy.executors.MultiThreadedExecutor(num_threads=4)
    executor.add_node(node)
    try:
        executor.spin()
    finally:
        node.destroy_node()
        rclpy.shutdown()
与 task_manager 的 main() 对比
对比项task_managercleaning_bot_llm
执行器rclpy.spin(node)（单线程）MultiThreadedExecutor(4)
线程数14
原因纯回调，无阻塞操作Service 调用可能阻塞，需要并发处理
为什么需要多线程执行器？goto_service() 调用后用 while 循环等待服务端返回，会阻塞当前线程。如果没有多线程，阻塞期间其他回调（如取消指令）就无法处理了。4 个线程确保 Service 等待、取消监听、状态更新可以并发进行。

十一、核心要点总结

关键认知
  • LLM 出代码，节点来执行。不同于传统机器人写死行为树，这里 LLM 实时生成 Python 代码并 exec() 执行，实现了自然语言到机器人动作的转化。
  • task_manager 的"干活的手"。它不规划任务，只执行分配给自己的任务。task_manager 说"去清洁"，它就调用 GPT-4o 生成清洁代码并运行。
  • Service 调用是可取消的。通过协作式取消标志位 + 循环中持续检测，实现了安全的任务中断。
  • 多线程执行器是必然选择。因为 Service 调用是阻塞的，必须用 MultiThreadedExecutor 让其他回调（尤其是取消指令）有执行机会。
  • 受限的沙箱执行。LLM 生成的代码在 exec() 中运行时，只有 node 和 TaskCancelledException 可用，内置函数被禁用。
  • 与 task_manager 的职责分工明确：task_manager 用 LLM 做总体规划（谁做什么），cleaning_bot_llm 用 LLM 做具体执行（怎么走到那个位置）。
