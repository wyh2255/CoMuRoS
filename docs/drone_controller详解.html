为什么无人机需要 PID 而全向机器人只需要 P？
  1. 飞行器动力学更复杂：四旋翼是欠驱动系统（6 自由度、4 个输入），有复杂的空气动力学效应，P 控制不够
  2. 积分项消除静差：无人机受重力影响，Z 方向需要 I 项消除稳态误差
  3. 微分项提供阻尼：飞行器容易震荡，D 项提供预测性阻尼
  4. 三步策略避免耦合：位置控制和偏航控制在四旋翼上耦合紧密，分步执行更稳定

三、PIDController 类（第 35-95 行）

文件首先定义了一个通用 PID 控制器类，采用位置式 PID 算法：

class PIDController:
    def __init__(self, kp, ki, kd, output_limit=1.0, integral_limit=0.5):
        self.kp = kp; self.ki = ki; self.kd = kd
        self.output_limit = output_limit
        self.integral_limit = integral_limit
        self.integral = 0.0
        self.prev_error = 0.0

    def compute(self, error, dt):
        p_term = self.kp * error                                  # P = Kp * e
        self.integral += error * dt                              # 累积积分
        self.integral = max(min(self.integral, self.integral_limit), -self.integral_limit)  # 抗积分饱和
        i_term = self.ki * self.integral                          # I = Ki * integral
        if dt > 0:
            d_term = self.kd * (error - self.prev_error) / dt      # D = Kd * de/dt
        else:
            d_term = 0.0
        self.prev_error = error
        output = p_term + i_term + d_term
        output = max(min(output, self.output_limit), -self.output_limit)
        return output

    def reset(self):  # 清零积分项和上一次偏差
        self.integral = 0.0; self.prev_error = 0.0

三个关键设计

特性实现目的
抗积分饱和 integral_limit = 0.5 限制积分项累积量，防止积分饱和导致的超调
输出限幅 output_limit = 1.0 限制最大控制输出（对应最大速度 1.0 m/s）
dt 保护 if dt > 0 首次循环 dt=0 时跳过微分项，避免除零错误
位置式 PID vs 增量式 PID：这里使用的是位置式 PID（完整计算 P+I+D）。相比增量式（只计算 Δu），位置式有积分累积的历史记忆，适用于连续位置控制场景。但需要抗积分饱和来防止过大的积分累积。

4.1 独立的四轴 PID 参数

无人机为四个自由度（X, Y, Z, Yaw）分别配置了不同的 PID 增益：

轴KpKiKdOutput LimitIntegral Limit设计思路
X / Y（水平）2.50.10.81.01.0高比例增益快速响应，微分阻尼防震荡
Z（高度）2.50.20.61.01.0更高积分增益消除重力导致的静差
Yaw（偏航）1.00.10.51.0—较低增益避免偏航震荡
参数设计的物理含义：
Z 轴的 Ki 是 X/Y 的 2 倍（0.2 vs 0.1）——因为无人机需要对抗重力，积分项必须更强才能消除高度静差。
Yaw 轴的 Kp 只有 X/Y 的 40%（1.0 vs 2.5）——偏航控制增益过高会导致无人机在目标方向附近来回摆动（震荡）。

五、三步递进式服务回调（第 225-308 行）

goto_service_callback() 是控制器的大脑，采用三步递进控制策略：

STEP 1
移动到位置
只控 X, Y, Z
偏航角速度 = 0
→
STEP 2
修正偏航角
锁定当前位置
只旋转到目标朝向
→
STEP 3
精细微调
同时控制四轴
达到最终精度
为什么分三步？四旋翼无人机在偏航旋转时，水平方向的推力方向会随之改变（因为电机固定在机体上）。如果在旋转的同时还要保持精确位置，控制会非常复杂。分步执行的哲学：一次只解决一个问题。 
def goto_service_callback(self, request, response):
    goal_x = request.x; goal_y = request.y; goal_z = request.z
    goal_yaw = math.radians(request.yaw_deg)

    self.cancel_requested = False
    # 重置所有 PID（关键！否则积分项会残留上一轮的值）
    self.pid_x.reset(); self.pid_y.reset()
    self.pid_z.reset(); self.pid_yaw.reset()

    # 步骤 1
    result = self.move_to_position(goal_x, goal_y, goal_z, timeout=400000005.0)
    if not result["success"]: return response

    # 步骤 2
    result = self.correct_yaw(goal_yaw, timeout=10000100005.0)
    if not result["success"]: return response

    # 步骤 3
    result = self.fine_tune_both(goal_x, goal_y, goal_z, goal_yaw, timeout=20000000000.0)
    if not result["success"]: return response

    self.stop()
    response.success = True
    response.message = "Drone reached goal position and yaw!"
    return response
注意：超时时间——步骤 1 的超时约 400,000,005 秒（> 12 年），步骤 2 约 10,000,100,005 秒（> 300 年），步骤 3 约 20,000,000,000 秒（> 600 年）。这些超时毫无实际意义，是全向控制器遗留的"无限等待"模式。上层（drone_llm.py）通过取消机制来保证超时控制。

六、Step 1: move_to_position（第 311-376 行）

目标：将无人机移动到指定的 (goal_x, goal_y, goal_z) 位置，不控制偏航。

def move_to_position(self, goal_x, goal_y, goal_z, timeout=45.0):
    timeout_time = time.time() + timeout
    last_time = time.time()

    while rclpy.ok():
        if time.time() > timeout_time: return {"success": False, ...}
        if self.cancel_requested: return {"success": False, ...}

        now = time.time(); dt = now - last_time; last_time = now
        if dt <= 0: dt = 0.01

        # 计算世界坐标系下的误差
        ex = goal_x - self.x; ey = goal_y - self.y; ez = goal_z - self.z

        # PID 控制（世界坐标系）
        vx_world = self.pid_x.compute(ex, dt)
        vy_world = self.pid_y.compute(ey, dt)
        vz = self.pid_z.compute(ez, dt)

        # 世界 → 机体 旋转变换
        vx_body = vx_world * math.cos(self.yaw) + vy_world * math.sin(self.yaw)
        vy_body = -vx_world * math.sin(self.yaw) + vy_world * math.cos(self.yaw)

        cmd = Twist()
        cmd.linear.x = vx_body; cmd.linear.y = vy_body
        cmd.linear.z = vz; cmd.angular.z = 0.0
        self.cmd_pub.publish(cmd)

        if abs(ex) < 0.05 and abs(ey) < 0.05 and abs(ez) < 0.05:
            return {"success": True, ...}

        time.sleep(0.05)

本步骤要点

  • 偏航角速度 = 0：步骤 1 不关心朝向，全力保证位置到达
  • 返回字典格式：使用 {"success": bool, "message": str} 结构，比单纯 bool 提供了更丰富的诊断信息
  • dt 实时计算：每个循环计算实际时间步长，用于 PID 中的微分项和积分项

七、Step 2: correct_yaw（第 379-452 行）

目标：锁定当前位置，将无人机朝向旋转到目标偏航角。

def correct_yaw(self, goal_yaw, timeout=15.0):

    target_x = self.x; target_y = self.y; target_z = self.z

    self.pid_x.reset(); self.pid_y.reset(); self.pid_z.reset()

        vy_world = self.pid_y.compute(ey, dt)
        vz = self.pid_z.compute(ez, dt)
        wz = self.pid_yaw.compute(eyaw, dt) * 0.5  # ← 增益减半防震荡

        # 同样做旋转变换
        ...

        if abs(eyaw) < 0.05: return {"success": True, "message": "Yaw corrected"}

两个关键细节

细节 1：锁定当前位置

进入步骤 2 时，快照当前坐标作为保持目标（target_x/y/z = self.x/y/z）。这样位置控制的误差始终是"当前位置 — 上一秒位置"，相当于命令无人机"就在这儿悬停，只转方向"。

同时位置 PID 被重置（reset()），因为 target 突然变化了，旧的积分值不再适用。

细节 2：偏航增益减半

wz = self.pid_yaw.compute(eyaw, dt) * 0.5

直接将 PID 计算出的偏航角速度乘以 0.5，进一步降低角速度，防止无人机在旋转过程中因位置扰动而震荡。这是一个经验性的安全措施。

三步策略的工程哲学

步骤控制内容PI D 重置收敛条件哲学
Step 1X, Y, Z（偏航=0）全部重置位置误差 < 0.05m先到附近再说
Step 2偏航（位置保持）位置重置，偏航保留偏航误差 < 0.05rad角度对准
Step 3X, Y, Z, 偏航全部重置四轴同时达标最后修一下
为什么 step 2 不重置偏航 PID？这是一个巧妙的设计。Step 1 中偏航 PID 没有被使用（angular.z = 0），所以它的积分项为 0，没有累积任何值。但如果在 step 1 中 已经 有了一些偏航控制，保留偏航积分意味着 Step 2 可以基于之前的控制趋势继续调整。

九、世界坐标系 → 机体坐标系变换（核心算法）

这三个步骤中都在做的核心运算：

# 世界坐标系下的速度（期望运动方向）
vx_world = self.pid_x.compute(ex, dt)    # 指向正东
vy_world = self.pid_y.compute(ey, dt)    # 指向正北

# 旋转变换到机体坐标系
vx_body = vx_world * cos(yaw) + vy_world * sin(yaw)
vy_body = -vx_world * sin(yaw) + vy_world * cos(yaw)
为什么需要这个变换？

/cmd_vel 的速度指令是机体坐标系下的：

  • linear.x = 无人机机头方向
  • linear.y = 无人机左侧方向
  • linear.z = 无人机正上方（始终向上，不受偏航影响）

但 PID 计算的误差 goal - current 是世界坐标系下的：

  • ex = 目标在 X 方向的距离（正向 = 东）
  • ey = 目标在 Y 方向的距离（正向 = 北）

如果无人机的机头朝东（yaw = 90°），世界坐标系的"向东走"就是机体坐标系的"向左走"。因此需要旋转矩阵将世界速度变换到机体速度。

旋转变换矩阵
[vx_body]    [ cos(yaw)   sin(yaw) ] [vx_world]
[vy_body] = [-sin(yaw)   cos(yaw) ] [vy_world]
示例：yaw = 90°（机头朝东）
世界：向东 1m/s → 机体：左移 1m/s
世界：向北 1m/s → 机体：前进 1m/s

十、start_drone() — 无人机启动（第 183-190 行）

def start_drone(self):
    msg = Twist()
    msg.angular.z = 0.1       # 发送微小偏航角速度
    self.cmd_pub.publish(msg)
    time.sleep(1)
    msg = Twist()               # 零速度停止
    self.cmd_pub.publish(msg)
用途：Gazebo 仿真中的无人机模型需要先收到一个非零的速度指令才能"激活"电机。这个微小扰动（angular.z=0.1，很小很安全）相当于踩一脚油门再松掉，让物理引擎开始计算旋翼动力学。没有这个步骤，无人机可能一直悬在空中不动。

全向控制器没有这个函数——因为地面机器人不需要"激活"，轮子天生就可以转动。

其他辅助方法

def stop(self):              # 发布零速度 Twist
    self.cmd_pub.publish(Twist())

@staticmethod
def wrap(a):               # 角度归一化 [-π, π]
    return math.atan2(math.sin(a), math.cos(a))

十一、关键知识点总结

从全向控制器到无人机控制器的进化
  • P → PID：从简单比例到完整的比例-积分-微分控制。积分项消除重力导致的静差，微分项提供阻尼防止震荡。
  • 一步 → 三步：从"直接走过去"到"先到位 → 再转向 → 后微调"。分步策略降低了多轴耦合控制的复杂性。
  • 2D → 3D：增加了 Z 轴（高度）控制，独立的 PID 参数（Ki 更高以对抗重力）。
  • 无变换 → 旋转变换：无人机悬停时可以任意朝向，必须将世界坐标系的速度指令转换到机体坐标系。
  • 无启动 → start_drone()：飞行器物理仿真需要激活步骤。
可改进之处
  • 超时时间荒谬：年级别的超时值没有实际意义，应改为根据目标距离动态计算
  • 缺少 gain scheduling：远处用大增益快速靠近，近处用小增益精确收敛。现在是一组参数走到底
  • none
  • 缺少 hover 模式：没有独立的"悬停"方法，取消任务后只是 stop()，无人机可能自由落体
  • 线程安全：self.x/y/z/yaw 在多线程下共享，没有显式加锁
  • Step 2 的偏航增益减半是硬编码：0.5 这个 magic number 应该作为可配置参数
与全向控制器共同的设计模式
  • 阻塞式服务回调 + ReentrantCallbackGroup + MultiThreadedExecutor
  • 协作式取消通过 cancel_requested 标志位
  • 比例控制 + 角度归一化 wrap()
  • 底层执行层：两个控制器都是 cleaning_bot_llm / drone_llm 的"手"，它们说去哪儿，控制器就去哪儿
  • 相同的 ROS 2 接口模式：odom 输入 + cancel 输入 + cmd_vel 输出 + goto_pose 服务
学习收获：同时读全向控制器和无人机控制器，我们看到了一次精彩的控制算法演进。当机器人从 2D 地面进入 3D 空中，控制器的复杂度几乎翻倍——新维度带来了重力补偿、坐标变换、多轴解耦等新问题。这正是机器人学中"移动平台决定控制策略"的生动体现。
