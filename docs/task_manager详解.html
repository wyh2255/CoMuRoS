task_manager.py 详解

CoMuRoS 系统的"大脑"——大语言模型驱动的任务调度中枢 · 1743 行

一、文件概览

task_manager.py 是整个 CoMuRoS 系统的任务调度中枢。 它不做底层机器人控制，而是充当"指挥官"角色：

一句话核心逻辑

订阅对话 → 调用 LLM 解析任务 → 发布结构化 JSON 给各机器人 LLM

重要认知：TaskManager 不直接控制机器人。它只负责"想清楚要做什么"，然后通过 /task_manager/tasks_json 话题把任务分配发布出去。具体的执行由各机器人 LLM 节点完成。

三、__init__ 初始化详解

__init__ 是节点的构造函数，在 ROS 节点创建时执行一次。按逻辑分为以下几个步骤：

节点命名。 通过 super().__init__("task_manager") 将节点注册为 task_manager。
读取 launch 参数。 
self.declare_parameter("config_file", "robot_config_roscon_2025")
self.declare_parameter("model", 10)
这两个值由 launch 文件指定。参数默认值配置在 .launch.py 文件中。
加载机器人配置 JSON。 
with open(cfg_path) as f:
    self.robot_config = json.load(f)
配置包含机器人名称、能力描述、形态参数、任务规则等，后续会被拼接到 LLM prompt 中。
初始化实例变量。 包括 LLM 客户端、任务队列、会话日志、状态标志位等。
创建 ROS 2 订阅与发布。 这是 ROS 2 节点的核心——告诉框架你关心哪些话题，别人发消息时会自动通知你（回调）。
设计思路：task_manager 不保存任务状态在文件或数据库中，所有状态都留在内存里。所以节点重启后对话历史和任务状态都会丢失——这是设计上的权衡（简单性优先于持久性）。

文件中有 4 个主要的回调函数，它们之间的调用关系：

↕

仿真时间更新
↕
回调 on_robot_state_callback

回调函数是事件驱动的——外部话题有新消息时，ROS 2 框架自动调用对应的回调。你不用手动轮询，rclpy.spin(node) 负责持续监听和分发。

注意：回调函数应当尽量轻量，避免长时间阻塞。因为 ROS 2 中 spin 是单线程的（默认 Executor），一个回调卡住了，所有其他回调都得排队等。

它动态拼接以下几部分：

# 系统提示词 ≈

"You are a Task Manager AI for a team of robots..."
+ 机器人能力描述
+ 机器人形态描述
+ 任务规则
+ 重规划规则

这个 prompt 的特点是：高度动态。机器人的能力、位置、状态、甚至仿真时间都是运行时动态填充的，让 LLM 能基于最新的上下文做决策。

五、LLM 调用 — parse_tasks_with_gpt

parse_tasks_with_gpt()（第544行）是整个文件的"引擎"。它的执行流程：

构建消息。 build_messages() 将对话历史转换为 LLM 格式的消息列表。
构建系统提示词。 build_system_prompt() 生成角色设定。
选择 LLM 后端。 根据 self.model 参数（0-100+）选择调用哪个模型。
发送请求。 调用对应后端的 API。
解析输出。 检测输出是否包含 "Plan:" 或 "Independent Tasks:"。
启动监控。 根据任务类型创建定时器，监控执行进度。

多 LLM 支持架构

文件支持 4 大类 LLM 后端（第572行起）：

后端model 编号范围调用函数示例模型
OpenAI 0-15 call_openai() GPT-4o, GPT-4.1, GPT-5.2, o1, o3-mini
Ollama 16-79 call_ollama() Gemma, DeepSeek, Qwen, Llama, Mistral
Gemini 80-85 call_gemini() Gemini 2.0 Pro/Flash, 2.5 Pro

模型选择机制：launch 文件通过 model:=10 参数传入一个数字，parse_tasks_with_gpt 根据这个数字走不同的 if/elif 分支。例如 model=10 调用 GPT-5.2，model=81 调用 Gemini 2.5 Pro。

OpenAI 调用的两种接口

文件中实现了两种 OpenAI 调用方式：

函数API适用模型
call_openai() openai.chat.completions.create() GPT-4 系列（GPT-4o, GPT-4.1 等）
call_openai_2() client.responses.create() GPT-5.x 系列（新 Responses API）

区别在于 OpenAI 在 GPT-5 系列引入了新的 Responses API，不再使用传统的 Chat Completions 接口。

六、任务解析 — extract_task_data

extract_task_data()（第1257行）解析 LLM 文本输出中的结构化数据。

LLM 返回的是一个自然语言文本，其中包含了结构化的任务列表。这段代码用正则表达式从中提取关键信息：

LLM 输出示例（Plan 模式）
Plan:
1. cleaning_bot: Go to the trash location
2. delivery_bot: Wait near the trash
3. cleaning_bot: Start cleaning
multirobot_list: [cleaning_bot, delivery_bot]
singlerobot_list: []
LLM 输出示例（Independent Tasks 模式）
Independent Tasks:
cleaning_bot tasks: clean floor, sweep area
delivery_bot tasks: go to charging station
multirobot_list: []
singlerobot_list: [cleaning_bot, delivery_bot]

extract_task_data 会从文本中提取三个关键数据结构：

变量类型含义
multirobot_listlist[str]参与多机器人顺序任务的机器人名称列表
singlerobot_listlist[str]参与独立任务的机器人名称列表
single_tasks_dictdict[str, str]独立任务中每个机器人的具体任务描述
脆弱点：这种方法依赖于 LLM 严格遵守输出格式。如果 LLM 的格式稍有偏差（比如漏写了 colon、使用了不同的大小写），正则匹配就可能失败。这是基于 LLM 的系统中常见的"输出格式不一致"问题。

七、任务发布机制

从 LLM 解析出任务分配后，task_manager 根据任务类型采用不同的发布策略：

1. Plan（顺序任务）— 队列逐个发布

create_queue()
解析 Plan 文本 → FIFO 队列
→
publish_single_task()
取出队首任务 → 发布 JSON
→
monitor_task_progress()
定时器等待 COMPLETE
→
publish_single_task()
完成 → 取下一个任务

核心逻辑在 publish_single_task()（第1129行）：

def publish_single_task(self, ai_output):
    task = self.task_queue.get()           # 从队列取一个任务
    robot_name, desc = task.split(": ", 1) # 解析"机器人: 任务"格式

    self.robot_tasks = {robot_name: desc}   # 构建任务字典

    json_obj = {
        "robot_tasks": self.robot_tasks,
        "sequence": task
    }

    json_msg = String()
    json_msg.data = json.dumps(json_obj)
    self.pub_tasks_json.publish(json_msg)   # 发布到话题

2. Independent Tasks（并行任务）— 批量发布

独立任务因为没有先后依赖关系，直接一次批量发布：

update_all_robot_tasks()
批量提取所有独立任务
→
publish_plan_json("Normal")
一次发布全部任务 JSON

每个机器人 LLM 节点订阅了 /task_manager/tasks_json，收到 JSON 后自动解析并执行。

八、进度监控与事件驱动重规划

task_manager 通过 ROS 2 定时器实现进度监控。

定时器机制

# 创建定时器，每 0.1 秒检查一次任务进度
self.plan_timer = self.create_timer(
    0.1,
    lambda: self.monitor_task_progress(ai_output)
)

monitor_task_progress()（第1076行）的执行逻辑：

def monitor_task_progress(self, ai_output):
    if "COMPLETE" in self.task_status:
        robot_name = extract_robot_name(self.task_status)
        self.publish_single_task(ai_output)  # 发布下一个任务

工作流程：

发布第一个任务。 Plan 中的 1 号任务被发布到 /task_manager/tasks_json。
机器人执行。 对应的机器人 LLM 收到任务，调用 ROS 2 Service 控制仿真中的机器人移动。
发布完成状态。 机器人执行完毕后，在 /chat/task_status 上发布 "COMPLETE" 消息。
定时器检测到 COMPLETE。 monitor_task_progress 被触发，发布 Plan 中的下一个任务。

事件驱动重规划

当用户或系统发送包含 "Event:" 前缀的消息时，task_manager 触发全量重规划：

事件类型行为
Event: RELEVANT（影响当前任务）停止受影响的机器人 → 重新规划 → 恢复未完成的任务 → 分配新任务
Event: IRRELEVANT（不影响任务）忽略事件，不触发重规划

重规划逻辑完全交给 LLM 判断——prompt 中告知 LLM 事件内容和当前任务状态，由 LLM 决定是否需要重新规划和如何调整。

九、main 入口

main() 函数（第1728行）是所有 ROS 2 节点的标准入口模式：

def main():
    rclpy.init()                     # 1. 初始化 ROS 2 通信层
    node = TaskManager()             # 2. 创建节点实例（自动注册所有订阅/发布）
    try:
        rclpy.spin(node)             # 3. 开始事件循环（阻塞，持续处理回调）
    except KeyboardInterrupt:
        node.get_logger().info(...)
    finally:
        node.destroy_node()
        rclpy.shutdown()
与 GUI 节点的区别

task_manager 不需要后台线程，因为它没有 GUI。 rclpy.spin(node) 直接在主线程中阻塞，通过回调机制处理所有消息。 需要 GUI 的节点（如 chat_gui.py）才需要在后台线程 spin，让主线程运行 customtkinter 事件循环。

十、核心要点总结

关键认知
  • TaskManager = 指挥官，不是执行者。它用 LLM 规划任务，用 ROS 2 话题分发任务，但不控制机器人移动。
  • 事件驱动。收到人类消息 → 调用 LLM → 发布 JSON。收到 COMPLETE → 发下一个任务。收到 Event → 重规划。
  • 双任务模式。Plan（顺序执行）和 Independent Tasks（并行执行），两种模式可以共存。
  • 状态都在内存。对话历史、机器人状态、任务队列——全存在 Python 对象中，重启即丢失。
  • 多 LLM 后端。同一套代码适配 OpenAI、Gemini、xAI、Ollama，通过 model 参数切换。
  • 核心弱点。高度依赖 LLM 的输出格式一致性，正则解析方式比较脆弱。
